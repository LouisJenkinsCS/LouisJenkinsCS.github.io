9 /*
10  * @Author: Louis Jenkins
11  * @Version: 1.0
12  * @Brief: String_Utils Garbage Collector
13  *
14  * String_Utils_Garbage_Collector, which I was denote as SU_GC, is an experiment of mine
15  * where it will attempt to make strings allocated from String_Utils functions as self-managed
16  * as possible. Unfortunately, it's not fully implemented, and might not be for a while, but
17  * I do have my sights on a goal. Currently implemented features will be listed below...
18  *
19  * 1) Very simple and basic virtual machine that, so far, has a stack for any allocated
20  * temporary variables pushed on it.
21  * 1A) Uses a Mark-And-Sweep algorithm to determine what should be freed.
22  *
23  * Eventually going to be implemented features:
24  *
25  * 2) A heap to allocate more permanent long-term strings on it.
26  * 2A) Based on a binary heap (tree) with key/value pairing for each to find and traverse nodes.
27  * 2B) Will be allocated on this heap with a custom MALLOC (SU_MALLOC).
28  * 2C) Will keep track of everything allocated, eliminating memory leaks.
29  * 3) Multi-threaded SU_GC and thread-safety
30  * 3A) SU_GC will run on it's own thread, waiting until a certain condition is met to algorithmically mark-and-sweep.
31  * 3B) Will be a thread-safe operation, with each thread being able to either share a virtual machine, or get it's own.
32  *
33  * Benefits of SU_GC (Unproven)
34  *
35  * 1) Easier to manage strings allocated, leaving behind the possibility of memory leaks.
36  * 1A) Temporary strings allocated in functions are pushed on a stack, then popped off after. The
37  * benefit of this being that SU_GC does not free them immediately, saving resources
38  * and time by doing so at a later date (In it's own thread [Not implemented])
39  * 1B) Strings allocated on the heap have a 0% chance of being leaked when the application exits, even
40  * if the string's reference is lost without being freed. So, every string will be freed when the
41  * virtual machine shuts down, regardless of where in your program you initialized it.
42  * 2) High performance with no possibility of hang ups.
43  * 2A) Each instance of the SU_GC will run on it's own thread, separate from current thread
44  * in use [Not Implemented].
45  * 2B) SU_GC waits until a condition variable is set to collect objects on the stack and heap.
46  * 3) Educational!
47  * 3A) This is purely for me, the creator, I never thought I'd ever implement something like SU_GC,
48  * but it is, needless to say, a very educational experience.
49  */
