1 /*
2  * @Author: Louis Jenkins
3  * @Version: 1.1
4  *
5  * What is String_Utils? String_Utils is basically an attempt at implementing a
6  * very useful, somewhat efficient String library with basic string manipulations
7  * comparators, and utilities offered in object oriented languages. In fact, this
8  * is based on Java's String object's methods, hence the name of most of the functions.
9  *
10  * My reason for creating this is, not just for fun, and boy was it ever, but
11  * also because I haven't found any attempt at creating a string library the way
12  * I did. I rather dislike the way C's String library handles thing, it's too
13  * minimal with how it abstracts and encapsulates it's functions, and plus it
14  * doesn't even have the basic functions that most people use day-to-day, but they
15  * do however give you the tools to do it yourself, so I decided to.
16  *
17  * String_Utils has a plethora of well-tested (as well as you can with one person
18  * writing this in one week) functions that you've grown to love in OOP languages
19  * like Java, I.E Substring, Index_OF, Split/Join, etc. I attempted to implement
20  * them as closely as they would be in Java, although of course since C and Java
21  * are vastly different languages, with different paradigms, it's impossible
22  * to make it exactly like so.
23  *
24  * Another nice feature is the passing of parameters to tailor the operation the way you want
25  * it if the library function supports it. For instance, one of the problems I had with the standard
26  * C library was that strcat modifies the original string AND returns the pointer to the same
27  * string it modified. I figured, why not just have an option to modify the string passed or not?
28  * What if I want to work with a string literal? Then with strcat, it would segfault. With my
29  * concat implementation, you can pass a string literal and not have it attempt to modify it, instead
30  * create a copy of the string for you. In some functions, multiple parameter passing can be passed
31  * with the | operator, I.E 'MODIFY | IGNORE_CASE'
32  *
33  * Another note, to keep the feature of being able to modify the string, all functions that can modify,
34  * even if you do not modify it directly, must pass the address of the string (& operator). For exmaple,
35  * String_Utils_concat(&string_one, string_two, MODIFY). A minor inconvenience.
36  *
37  * I hope you enjoy my first project as much as I will, I worked very hard on it.
38  * Hope it shows! Enjoy!
39  */

44 /*
45  * General Notes: More than one parameter can be passed to any given function,
46  * but of course, they must make sense. If you pass a function which supports it both
47  * LOWERCASE and UPPERCASE it'll just be the same string you ended up with before
48  * passing the parameters. Also, passing nonsensical parameters aren't likely to do anything
49  * like NONE | REVERSE will just REVERSE as NONE does nothing. The parameters are
50  * very basic and bare bones, but they will be (most likely) developed into something
51  * more full fledged.
52 */
